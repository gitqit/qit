## Learned User Preferences

- When implementing an attached plan, do not edit the plan file; use existing todos only (mark items in progress, do not recreate the todo list).
- For this workspace’s tooling, prefer a minimal public workflow: single command with path argument, ngrok started automatically, and the tunnel URL shown in output; keep `qit serve` startup output scannable (avoid large banners and dumping internal paths or snapshot metadata unless debugging).
- For `qit` CLI surface, prefer Git command parity (familiar subcommands and flags) over custom verb names where it matches user expectations.
- For `qit` auth UX, show per-session username and password in the terminal by default and include them in suggested `git clone` commands; offer a flag such as `--hidden-pass` when obscuring credentials is preferred.
- For the `qit` Web UI, only localhost may skip the login gate; exposed transports (including Tailscale, ngrok, and LAN) must require Web UI authentication.
- When starting `qit`, show the local Web UI URL in the terminal so operators can open the session that uses the localhost bypass.
- For `qit-webui` Code views, follow GitHub-style browsing: combine the file tree with the main content, render a folder’s README when one exists (not only at repo root), keep folder navigation in the tree workflow, and avoid showing full local filesystem directory names—use repository-relative presentation; for diffs prefer a unified single-pane presentation (removed lines above added lines with clear red/green emphasis) over side-by-side; for file contents in the code editor, prefer opening plain-text raw in a new browser tab over clipboard-only copy.
- Prefer compact, icon-forward UI with a strong React icon set and clear file/folder affordances; keep radii and core `Button` padding on shared design tokens rather than one-off values, place primary repo actions (for example create branch / open pull request) on the same row as the tab navigation to limit layout shift, and keep toast surfaces opaque (avoid heavy transparency on notifications).
- In user-facing branding and documentation, spell the product **Qit** (capital Q), not all-lowercase.
- For pull requests, target GitHub-style workflows and behavior in both the Web UI and CLI; shape PR-related CLI commands after **GitHub CLI** (`gh`) patterns where that matches user expectations.
- For Web UI pull request comments and reviews, support a per-browser display name for the commenter and persist it in **localStorage** (not only server-side identity).
- Keep **landing** and **qit-webui** visual branding aligned: reuse the same logo assets and apply theme-appropriate variants (for example, a logo intended for dark backgrounds on light UI themes and vice versa).

## Learned Workspace Facts

- Cargo workspace `qit` (“Quick Git”): CLI in `qit-cli`; web UI (Vite/React/TypeScript, Tailwind CSS 4, Headless UI, atomic-design layout) in `crates/qit-webui`; sidecar and worktree behavior in `crates/qit-git`; orchestration and `RepoStore` boundaries in `crates/qit-domain`; HTTP smart protocol and transports (local, LAN, ngrok, Tailscale) in `crates/qit-http` and `crates/qit-transports`; registry/persistence in `crates/qit-storage`; integration tests under `tests/`.
- The `landing/` app is the GitHub Pages marketing and documentation site (Vite, React, TypeScript, Tailwind CSS 4, Headless UI, atomic design); primary docs are authored as **MDX** with Docusaurus-style frontmatter metadata for navigation and page titles; treat fenced code blocks as rich doc components (preview affordances, copy, captions where helpful), use syntax highlighting with a cohesive chalkboard/pastel palette, keep the docs sidebar fixed while the article column scrolls, and style inline code and links consistently with that system. Deploy the built site with `.github/workflows/landing-pages.yml` (GitHub Pages).
- The `qit-webui` SPA is served on the same origin as Git smart HTTP (alongside the repository URL path).
- Serving a folder that is already a Git worktree root (it contains `.git`) requires `--allow-existing-git`; snapshots skip paths that include a normal component named `.git` anywhere in the relative path (not only the top-level metadata directory), because libgit2 rejects such index entries; a first-time serve can infer the checked-out branch from the host repository.
- Public clone URLs include a path segment tied to the served folder name so default `git clone` checkout directory names stay readable; the worktree checked-out branch can differ from the exported (served) branch on the sidecar so local work can track one branch while Git clients still receive another (for example serving `main` while developing on a feature branch).
- Release binaries are expected to cover a wide platform matrix, including Linux ARM64, Windows ARM64, and macOS x64 (Intel), in addition to other primary targets. Tagged releases can update a Homebrew tap when repository variable `HOMEBREW_TAP_REPOSITORY` and secret `HOMEBREW_TAP_TOKEN` are set; the formula template lives under `packaging/homebrew/`.
- Web UI implicit-owner bypass is based on the request `Host` header (`localhost`, `127.0.0.1`, `::1`), not the client IP alone, so tunnel-forwarded requests that still carry a public or tailnet hostname require Web UI login; clone and copy affordances should show the session’s public base URL when one applies (LAN, ngrok, Tailscale), not only loopback addresses. CLI `--transport lan` binds the supervisor edge listener on `0.0.0.0` and advertises a LAN-reachable HTTP base URL from a detected non-loopback private IPv4; Web UI auth rules remain Host-header based.
- Custom Cursor skills in this repository live under `.cursor/skills/<name>/` as a concise `SKILL.md` (frontmatter + instructions) plus a companion `REPORT_TEMPLATE.md` for the full output shape; examples include `web-product-audit`, `security-code-audit`, `cli-product-review`, `codebase-engineering-review`, and `test-qa-risk-audit`.
- For `qit-webui` pull request detail (especially merged PRs), file diffs should reflect the PR’s recorded source/target commit range at merge time, not the live diff between branch tips that may have moved since.
- Multiple `qit serve` instances are intended to cooperate behind one shared local/LAN/ngrok/Tailscale entrypoint, exposing different repositories on distinct URL path prefixes rather than each owning a separate public base URL.
- For `qit-webui` development, Vite can be run with HMR by setting `QIT_DEV_API` and `QIT_DEV_MOUNT` to proxy API and Git requests to a running `qit serve` instance; release binaries embed `crates/qit-webui/frontend/dist/` at compile time (`include_bytes!`), so after changing Web UI source run `npm run build` in `crates/qit-webui/frontend` before `cargo build` when you need the embedded bundle to match (unless you rely on the dev proxy only).
- Access-request approval is completed in the requester’s browser via `completeOnboarding` using the `qit_request…` receipt; one-time `qit_setup…` codes from **Issue setup code** / **Reset setup** are for manual owner-driven onboarding, not the default post-approval path for requesters.
